## 1. 权限模型（五级权限）

### 1.1 权限级别定义

claw-code 实现了五级权限模式，位于 `rust/crates/runtime/src/permissions.rs`:

```rust
pub enum PermissionMode {
    ReadOnly,           // 只读模式
    WorkspaceWrite,     // 工作区写模式
    DangerFullAccess,   // 危险完全访问
    Prompt,             // 交互确认模式
    Allow,              // 完全允许
}
```

**权限层级关系**（由 Ord trait 确定）:
```
Allow > DangerFullAccess > Prompt > WorkspaceWrite > ReadOnly
```

### 1.2 权限模式别名解析

在 `rust/crates/runtime/src/config.rs` 中定义了模式别名映射：

| 配置值 | 解析为 |
|--------|--------|
| `"default"`, `"plan"`, `"read-only"` | `ReadOnly` |
| `"acceptEdits"`, `"auto"`, `"workspace-write"` | `WorkspaceWrite` |
| `"dontAsk"`, `"danger-full-access"` | `DangerFullAccess` |

### 1.3 工具级权限要求

通过 `PermissionPolicy.with_tool_requirement()` 为每个工具绑定最小权限级别：

```rust
// 示例配置
PermissionPolicy::new(PermissionMode::WorkspaceWrite)
    .with_tool_requirement("read_file", PermissionMode::ReadOnly)
    .with_tool_requirement("write_file", PermissionMode::WorkspaceWrite)
    .with_tool_requirement("bash", PermissionMode::DangerFullAccess)
```

### 1.4 权限规则（Allow/Deny/Ask）

在 `RuntimePermissionRuleConfig` 中定义三类规则：

- **allow 规则**: 强制允许特定工具/输入组合
- **deny 规则**: 强制拒绝特定工具/输入组合（优先级最高）
- **ask 规则**: 即使 ActiveMode 允许，仍需用户确认

规则语法支持精细化匹配：
```
tool_name              → 匹配所有输入
tool_name(subject)     → 精确匹配输入中的 subject 字段
tool_name(prefix:*)    → 前缀匹配
```

示例：
```
allow:   ["bash(git:*)", "read_file"]
deny:    ["bash(rm -rf:*)"]
ask:     ["Edit", "bash(npm:*"]
```

### 1.5 Hook 权限覆盖

Hooks 可在运行时通过 `PermissionContext` 覆盖权限决策：

```rust
pub enum PermissionOverride {
    Allow,   // 强制允许
    Deny,    // 强制拒绝
    Ask,     // 强制触发交互确认
}
```

## 5. 安全验证流程

### 5.1 工具执行前的权限检查

位于 `rust/crates/runtime/src/permission_enforcer.rs`:

```rust
pub struct PermissionEnforcer {
    policy: PermissionPolicy,
}

pub fn check(&self, tool_name: &str, input: &str) -> EnforcementResult {
    let outcome = self.policy.authorize(tool_name, input, None);
    match outcome {
        PermissionOutcome::Allow => EnforcementResult::Allowed,
        PermissionOutcome::Deny { reason } => EnforcementResult::Denied { ... },
    }
}
```

### 5.2 Hook 验证流程

位于 `rust/crates/runtime/src/hooks.rs`:

**PreToolUse Hook** 在工具执行前运行：
```rust
pub fn run_pre_tool_use(&self, tool_name: &str, tool_input: &str) -> HookRunResult
```

Hook 返回值决定是否继续：
- **exit 0**: 允许执行
- **exit 2**: 拒绝执行
- **其他非零**: 失败但不阻断

**Hook 输出协议**：
```json
{
  "systemMessage": "用户可见消息",
  "reason": "拒绝原因",
  "hookSpecificOutput": {
    "permissionDecision": "allow|deny|ask",
    "permissionDecisionReason": "原因",
    "updatedInput": { ... }
  }
}
```

### 5.3 三类 Hook 事件

| 事件 | 时机 | 典型用途 |
|------|------|----------|
| `PreToolUse` | 工具执行前 | 权限检查、输入修改 |
| `PostToolUse` | 工具执行后 | 结果验证、审计日志 |
| `PostToolUseFailure` | 工具失败后 | 错误处理、重试逻辑 |

### 5.4 文件写入检查

```rust
pub fn check_file_write(&self, path: &str, workspace_root: &str) -> EnforcementResult {
    match mode {
        PermissionMode::ReadOnly => Deny("file writes not allowed"),
        PermissionMode::WorkspaceWrite => {
            if is_within_workspace(path, workspace_root) {
                Allow
            } else {
                Deny("path outside workspace")
            }
        }
        PermissionMode::Allow | DangerFullAccess => Allow,
        ...
    }
}
```

### 5.5 Bash 命令检查

```rust
pub fn check_bash(&self, command: &str) -> EnforcementResult {
    match mode {
        PermissionMode::ReadOnly => {
            if is_read_only_command(command) {
                Allow
            } else {
                Deny("command may modify state")
            }
        }
        PermissionMode::Prompt => Deny("requires confirmation"),
        _ => Allow,
    }
}
```

## 7. 安全特性总结

| 特性 | 实现位置 | 描述 |
|------|----------|------|
| **五级权限模型** | `permissions.rs` | ReadOnly → WorkspaceWrite → DangerFullAccess → Prompt → Allow |
| **工具级权限绑定** | `permissions.rs` | 每个工具可配置最小所需权限级别 |
| **规则引擎** | `permissions.rs` | Allow/Deny/Ask 三类规则，支持精确/前缀匹配 |
| **Hook 覆盖** | `hooks.rs` | PreToolUse/PostToolUse/PostToolUseFailure 全生命周期 |
| **沙箱隔离** | `sandbox.rs` | Linux namespace + filesystem isolation |
| **只读命令白名单** | `permission_enforcer.rs` | 保守估计 60+ 只读命令 |
| **工作区边界检查** | `permission_enforcer.rs` | WorkspaceWrite 模式限制文件在 workspace 内 |
| **MCP 状态机** | `mcp_tool_bridge.rs` | Disconnected → Connecting → Connected/AuthRequired/Error |
| **OAuth 认证** | `mcp_client.rs` | 支持 MCP 服务器 OAuth 认证 |
| **配置合并** | `config.rs` | User → Project → Local 三级配置叠加 |
| **Python Port** | `src/permissions.py` | 工具拒绝名/前缀白名单机制 |

## 8. 安全建议

1. **Bash 命令白名单**: `is_read_only_command` 使用硬编码列表，建议改为配置驱动
2. **沙箱 fallback**: 非 Linux 系统完全无沙箱保护，需明确警告用户
3. **Hook 安全**: Hook 以当前用户权限运行，需确保 Hook 脚本可信
4. **MCP OAuth**: OAuth token 存储和传输需额外加密保护
5. **工作区逃逸**: WorkspaceWrite 模式仅做字符串前缀检查，可被符号链接绕过

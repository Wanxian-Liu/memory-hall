## 一、工具定义格式（ToolSpec）

### 1.1 核心数据结构

claw-code 的工具定义采用 `ToolSpec` 结构体，定义在 `rust/crates/tools/src/lib.rs`:

```rust
pub struct ToolSpec {
    pub name: &'static str,           // 工具名称，如 "bash", "read_file"
    pub description: &'static str,    // 工具描述
    pub input_schema: Value,          // JSON Schema 格式的输入参数定义
    pub required_permission: PermissionMode, // 执行所需的最低权限级别
}
```

### 1.2 工具来源分类

claw-code 的工具分为三层来源：

| 来源 | 说明 | 注册方式 |
|------|------|---------|
| **Built-in Tools** | Rust 内置的核心工具集，通过 `mvp_tool_specs()` 定义 | 编译时静态注册 |
| **Runtime Tools** | 运行时动态注册的工具 | `GlobalToolRegistry::with_runtime_tools()` |
| **Plugin Tools** | 插件提供的工具（如 MCP 工具） | `GlobalToolRegistry::with_plugin_tools()` |

### 1.3 核心内置工具列表

通过 `mvp_tool_specs()` 函数定义了 40+ 工具，分为几类：

**文件系统类**（WorkspaceWrite 或 ReadOnly）:
- `read_file` — 读取文件
- `write_file` — 写入文件
- `edit_file` — 编辑文件（替换文本）
- `glob_search` — glob 模式搜索
- `grep_search` — 正则搜索

**Shell 执行类**（DangerFullAccess）:
- `bash` — Bash 命令执行（最核心的工具）
- `PowerShell` — PowerShell 命令执行
- `REPL` — 交互式代码执行

**网络类**（ReadOnly）:
- `WebFetch` — URL 获取
- `WebSearch` — 网络搜索

**Agent/任务类**（DangerFullAccess）:
- `Agent` — 启动子代理
- `TaskCreate`/`TaskGet`/`TaskStop`/`TaskList` — 后台任务管理
- `TeamCreate`/`TeamDelete` — 团队管理
- `CronCreate`/`CronList`/`CronDelete` — 定时任务

**开发者工具**:
- `LSP` — 语言服务器协议查询
- `NotebookEdit` — Jupyter notebook 编辑
- `Config` — 配置读写
- `Skill` — 技能加载

**其他**:
- `TodoWrite` — 任务列表管理
- `Sleep` — 延时
- `SendUserMessage` / `Brief` — 消息发送
- `EnterPlanMode` / `ExitPlanMode` — 计划模式切换
- `StructuredOutput` — 结构化输出
- `MCP` — MCP 服务器工具调用
- `ListMcpResources` / `ReadMcpResource` — MCP 资源访问

## 二、工具注册与分发机制

### 2.1 GlobalToolRegistry

`GlobalToolRegistry` 是工具的中央注册表，管理三类工具的注册和分发：

```rust
pub struct GlobalToolRegistry {
    plugin_tools: Vec<PluginTool>,        // 插件工具
    runtime_tools: Vec<RuntimeToolDefinition>, // 运行时工具
    enforcer: Option<PermissionEnforcer>,  // 权限执行器
}
```

### 2.2 注册流程

1. **构建时注册**:
   ```rust
   let registry = GlobalToolRegistry::builtin()           // 创建空注册表
       .with_plugin_tools(plugin_tools)?   // 添加插件工具
       .with_runtime_tools(runtime_tools)? // 添加运行时工具
       .with_enforcer(enforcer);           // 设置权限执行器
   ```

2. **冲突检测**:
   - 插件工具名不能与内置工具名冲突
   - 运行时工具名不能与内置和插件工具冲突
   - 禁止重复的工具名

3. **工具名规范化**:
   ```rust
   fn normalize_tool_name(value: &str) -> String {
       value.trim().replace('-', "_").to_ascii_lowercase()
   }
   ```
   别名映射：
   - `read` → `read_file`
   - `write` → `write_file`
   - `edit` → `edit_file`
   - `glob` → `glob_search`
   - `grep` → `grep_search`

### 2.3 工具分发（allowedTools 过滤）

工具可以通过 `--allowedTools` 配置限制可用工具范围：

```rust
pub fn normalize_allowed_tools(&self, values: &[String]) -> Result<Option<BTreeSet<String>>, String>
```

调用链：
```
normalize_allowed_tools() 
  → 规范化每个工具名（去除空格/横杠、转小写）
  → 解析别名（read → read_file 等）
  → 验证工具名存在
  → 返回 BTreeSet<String>
```

### 2.4 工具定义导出

```rust
pub fn definitions(&self, allowed_tools: Option<&BTreeSet<String>>) -> Vec<ToolDefinition>
```

- 按 `allowed_tools` 过滤后返回 `ToolDefinition` 列表
- `ToolDefinition` 包含 `name`、`description`、`input_schema`，用于 LLM API 的 tool calling

## 三、权限控制模型

### 3.1 PermissionMode 枚举

```rust
pub enum PermissionMode {
    ReadOnly,          // 只读模式
    WorkspaceWrite,    // 工作区写入
    DangerFullAccess,  // 危险全访问
    Prompt,            // 需要交互确认
    Allow,             // 允许（特殊模式）
}
```

权限层级（从低到高）：`ReadOnly < WorkspaceWrite < DangerFullAccess`

### 3.2 PermissionPolicy

```rust
pub struct PermissionPolicy {
    active_mode: PermissionMode,              // 当前激活的权限模式
    tool_requirements: BTreeMap<String, PermissionMode>, // 工具→所需权限
}
```

### 3.3 工具权限要求映射

每个工具在 `mvp_tool_specs()` 中声明其所需权限：

| 工具 | 所需权限 | 说明 |
|------|---------|------|
| `bash` | DangerFullAccess | 危险：可执行任意命令 |
| `read_file` | ReadOnly | 只读文件访问 |
| `write_file` | WorkspaceWrite | 工作区写入 |
| `edit_file` | WorkspaceWrite | 工作区编辑 |
| `glob_search` | ReadOnly | 只读 |
| `grep_search` | ReadOnly | 只读 |
| `WebFetch` | ReadOnly | 只读网络请求 |
| `WebSearch` | ReadOnly | 只读网络请求 |
| `Agent` | DangerFullAccess | 可启动子代理 |
| `TaskCreate` | DangerFullAccess | 可创建后台任务 |
| `REPL` | DangerFullAccess | 可执行任意代码 |

### 3.4 权限检查流程

```
PermissionEnforcer.check(tool_name, input_json)
    ↓
PermissionPolicy.required_mode_for(tool_name)  // 查询工具所需权限
    ↓
比较 active_mode 与 required_mode
    ↓
if active_mode >= required_mode → Allow
else if active_mode == Prompt → 向用户请求确认
else → Deny
```

### 3.5 Hook 覆盖机制

```rust
pub enum PermissionOverride {
    Allow,   // Hook 强制允许
    Deny,    // Hook 强制拒绝
    Ask,     // Hook 要求询问
}
```

Hook 可以通过 `PermissionContext` 提供覆盖决策，在标准权限评估之前生效。

### 4.5 Bash 命令验证管道

claw-code 对 Bash 命令执行多层次验证：

```rust
validate_command(command, mode, workspace)
    ↓
┌─ 1. modeValidation ──────────────────────────────────────────┐
│   if mode == ReadOnly:                                        │
│     - 拦截写命令: cp, mv, rm, mkdir, touch, tee, truncate... │
│     - 拦截状态修改: apt, pip, npm, docker, systemctl...       │
│     - 拦截写重定向: >, >>, >&                                 │
│     - 拦截 git 写操作: push, pull, commit, merge...           │
│     - sudo 包裹的写命令也被拦截                               │
└───────────────────────────────────────────────────────────────┘
    ↓
┌─ 2. sedValidation ───────────────────────────────────────────┐
│   if mode == ReadOnly and sed -i: 拦截 sed in-place 编辑      │
└───────────────────────────────────────────────────────────────┘
    ↓
┌─ 3. destructiveCommandWarning ────────────────────────────────┐
│   检测危险命令模式:                                           │
│   - "rm -rf /" → 警告：递归强制删除根目录                    │
│   - "rm -rf ~" → 警告：删除主目录                            │
│   - "rm -rf *" → 警告：删除当前目录                           │
│   - "mkfs" → 警告：文件系统创建                              │
│   - "dd if=" → 警告：直接磁盘写入                            │
│   - ":(){ :|:& };:" → 警告：Fork 炸弹                        │
│   - "shred" → 警告：数据销毁                                 │
└───────────────────────────────────────────────────────────────┘
    ↓
┌─ 4. pathValidation ─────────────────────────────────────────┐
│   - 目录遍历检测: ../ → 如果不在 workspace 内则警告          │
│   - 主目录引用检测: ~/ 或 $HOME → 警告                        │
└───────────────────────────────────────────────────────────────┘
    ↓
返回 ValidationResult: Allow | Block { reason } | Warn { message }
```

## 五、工具执行流程

### 5.1 执行入口

```rust
pub fn execute(&self, name: &str, input: &Value) -> Result<String, String> {
    // 1. 先尝试内置工具
    if mvp_tool_specs().iter().any(|spec| spec.name == name) {
        return execute_tool_with_enforcer(self.enforcer.as_ref(), name, input);
    }
    // 2. 再尝试插件工具
    self.plugin_tools
        .iter()
        .find(|tool| tool.definition().name == name)
        .ok_or_else(|| format!("unsupported tool: {name}"))?
        .execute(input)
}
```

### 5.2 工具执行分发

```rust
fn execute_tool_with_enforcer(enforcer, name, input) -> Result<String, String> {
    match name {
        "bash" => {
            maybe_enforce_permission_check(enforcer, name, input)?;
            from_value::<BashCommandInput>(input).and_then(run_bash)
        }
        "read_file" => { /* ... */ }
        "write_file" => { /* ... */ }
        "edit_file" => { /* ... */ }
        // ... 所有其他工具
    }
}
```

### 5.3 权限检查时机

```rust
fn maybe_enforce_permission_check(enforcer, tool_name, input) -> Result<(), String> {
    if let Some(enforcer) = enforcer {
        enforce_permission_check(enforcer, tool_name, input)?;
    }
    Ok(())
}
```

注意：`maybe_enforce_permission_check` 仅对"需要权限检查"的工具调用。对于 `WebFetch`、`WebSearch` 等 ReadOnly 工具，**不进行运行时权限检查**（因为它们本身声明为 ReadOnly 权限）。

### 5.4 Bash 执行完整流程

```
run_bash(input: BashCommandInput)
    ↓
workspace_test_branch_preflight()  // 检查工作区测试前分支状态
    ↓
如果分支落后于 main 且命令是 cargo test:
    → 返回分支分歧警告，不执行命令
    ↓
如果通过预检:
    ↓
sandbox_status_for_input()  // 解析 SandboxConfig
    ↓
prepare_tokio_command()     // 准备 Tokio 命令
    ↓
如果 build_linux_sandbox_command() 返回 Some:
    → 使用 unshare 创建沙箱进程
    ↓
否则:
    → 直接用 sh -lc 执行
    ↓
如果指定了 timeout:
    → tokio::time::timeout() 包装
    ↓
执行命令，收集 stdout/stderr
    ↓
truncate_output()  // 16KB 截断
    ↓
返回 BashCommandOutput { stdout, stderr, sandbox_status, ... }
```

## 六、关键设计特点

### 6.1 安全性设计

1. **分层权限模型**: ReadOnly → WorkspaceWrite → DangerFullAccess
2. **沙箱隔离**: Linux namespace（user/mount/ipc/pid/uts/net）+ 文件系统模式控制
3. **命令验证管道**: 4 层验证（modeValidation → sedValidation → destructiveCommandWarning → pathValidation）
4. **输出截断**: 防止内存耗尽（16KB 限制）
5. **子代理工具白名单**: 严格限制子代理可用的工具

### 6.2 灵活性设计

1. **工具注册机制**: 内置 + 运行时 + 插件三层混合
2. **工具别名系统**: read → read_file 等用户友好的别名
3. **allowedTools 过滤**: 支持场景化工具限制
4. **SandboxConfig 可覆盖**: BashCommandInput 可覆盖全局沙箱配置
5. **Hook 权限覆盖**: 支持通过 Hook 动态调整权限

### 6.3 可观测性设计

1. **ToolExecution 追踪**: 记录工具执行状态和消息
2. **Branch Divergence 检测**: 在 workspace test 前检查 git 分支状态
3. **SandboxStatus 报告**: 每次 Bash 执行返回沙箱状态详情
4. **命令语义分类**: 可追踪命令的读写/网络/破坏性属性

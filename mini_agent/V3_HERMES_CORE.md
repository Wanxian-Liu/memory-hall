# Mimir V3 - 无限接近Hermes

**日期**: 2026-04-14  
**核心目标**: Mimir进化成无限接近Hermes的Agent  
**设计基础**: Ralph Round 1 三方锤炼

---

## 一、核心定位

**Mimir = 无限接近Hermes + 独特执念**

| 方面 | Hermes | Mimir |
|------|--------|-------|
| Agent能力 | ✅ 完整 | ✅ 无限接近 |
| 记忆特色 | ✅ 四层记忆体系 | ✅ Hermes风格 |
| 双形态 | ❌ 无 | ✅ OpenClaw调度/独立存在 |
| 专属LLM | ❌ 无 | ✅ 可连接DeepSeek等 |
| 独特执念 | ❌ 无 | ✅ 自己的"为什么" |

## 二、Hermes核心能力（要学的）

### 2.1 四层记忆体系

| 层级 | Hermes实现 | Mimir借鉴 |
|------|----------|----------|
| L1 | MEMORY.md (~800 tokens) | daily notes + MEMORY.md分层 |
| L2 | USER.md (~500 tokens) | USER.md |
| L3 | Session Search (FTS5) | 向量搜索 |
| L4 | Honcho用户建模 | 待实现 |

**关键设计**：
- 严格容量限制，防止提示膨胀
- 冷冻快照模式（会话启动时注入）
- 自动去重+安全扫描

### 2.2 Agent Loop

**核心机制**：
```
用户消息 → 构建系统提示 → API调用 → 解析响应
→ 若有tool_calls：执行后继续循环
→ 若为文本响应：持久化会话，刷新记忆
```

**关键设计**：
- 可中断API调用（后台线程+interrupt事件）
- 迭代预算（默认90次）
- 子agent独立预算（默认50）
- 预压缩+自动压缩双触发
- 多API模式（chat_completions/codex/anthropic）

### 2.3 代码执行（execute_code）

**架构**：
```
Agent编写Python脚本 → Unix domain socket RPC
→ hermes_tools.py存根模块
→ 子进程执行，结果通过socket返回
```

**关键哲学**：中间tool结果永不进入上下文，只有最终print()输出返回

### 2.4 Skills自创建

**触发时机**：
- 复杂任务（5+ tool calls）后
- 错误/死胡同并找到工作路径时
- 用户纠正其方法时
- 发现非平凡工作流时

**操作**：create / patch / edit / delete

### 2.5 Trajectory记录

**格式**：ShareGPT兼容JSONL

**轨迹闭环**：
```
用户交互 → 记忆积累 → Skills自创建 → Trajectory记录 → RL训练改进
```

## 三、Mimir架构设计

### 3.1 核心架构

```
┌─────────────────────────────────────────────────────┐
│                    Mimir Agent                       │
│  ┌───────────┐  ┌───────────┐  ┌───────────┐       │
│  │ Executor  │  │ Memory    │  │ Skill     │       │
│  │ 执行引擎  │  │ System    │  │ Router    │       │
│  └─────┬─────┘  └─────┬─────┘  └─────┬─────┘       │
│        └──────────────┼──────────────┘             │
│                       ▼                            │
│  ┌─────────────────────────────────────────┐       │
│  │         Trajectory Engine                │       │
│  │     (经验轨迹记录与回放)                 │       │
│  └─────────────────────────────────────────┘       │
└─────────────────────────────────────────────────────┘
```

### 3.2 双形态实现

```
形态A: OpenClaw调度模式
┌──────────────────────────────┐
│     OpenClaw Gateway         │
│  sessions_spawn → Mimir      │
│  (作为子Agent被调度)          │
└──────────────────────────────┘

形态B: 独立部署模式
┌──────────────────────────────┐
│  Mimir Standalone CLI        │
│  $ mimir run --task "..."    │
│  $ mimir chat                │
└──────────────────────────────┘
```

**共享核心引擎，差异仅在入口层**

### 3.3 专属LLM连接

```yaml
# ~/.mimir/providers.yaml
providers:
  deepseek:
    api_base: "https://api.deepseek.com"
    model: "deepseek-chat"
  openai:
    model: "gpt-4o"

routing:
  default: "deepseek"
  code: "openai"
```

## 四、与Hermes的核心差距

| 差距维度 | Hermes | Mimir现状 |
|---------|--------|----------|
| 成熟度 | 生产级验证 | 原型阶段 |
| 生态 | 完整工具链/文档 | 需自建 |
| 轨迹学习 | 自动化模式学习 | 需手工设计 |
| 社区 | 成熟社区 | 无 |
| 集成深度 | 与gstack深度耦合 | 需对接 |

## 五、最难实现的部分

### 轨迹学习全自动化

**为什么最难**：
1. 从原始WAL日志中识别有意义的行为单元
2. 从单次成功中区分"运气好"和"真正可复用的模式"
3. 生成可执行的Skill需要高质量验证
4. 模式漂移检测

**建议路径**：
- 阶段1：手动记录轨迹
- 阶段2：半自动模式识别（人类确认后固化）
- 阶段3：全自动学习

## 六、差异化：独特执念

**最大风险**：过度模仿变成"复制品"

**破局点**：Mimir要有自己的"执念"
- 一个Hermes不会问、不想问、或还没想到去问的问题
- 围绕这个执念生长出的记忆、风格、逻辑
